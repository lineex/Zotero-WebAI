# Deepseek Copliot v0.9.1 字体可读性设计

## 状态

已确认方向，属于一次聚焦可读性、且以宿主原生体验优先的小版本设计。

## 目标

将本次修改作为 `v0.9.1` 发布，聚焦解决 Deepseek Copliot 默认字体“略小、略挤、略不够原生”的使用反馈，让 Zotero 侧边栏中的主要阅读与输入体验更贴近 Zotero 自身的字体行为。

本次的目标不是单纯“把字调大一点”，而是尽量避免插件继续硬编码绝对 `px` 字号，改为优先继承 Zotero 宿主的字体与字体倍率，只在必要处用少量相对层级来表达标题、说明和标签之间的差异。

## 背景

已有用户反馈当前 Deepseek Copliot 的界面字体偏小，日常使用时阅读压力稍高。进一步调研后可以确认，当前问题不只是“默认字号稍小”，更是“插件自己的字体实现和 Zotero 宿主心智脱节”。

从现有实现来看，前端中分散存在大量硬编码的 `10px`、`11px`、`12px` 字号，主要分布在侧边栏、消息区、输入区和辅助提示区域。这带来三个问题：

- 默认阅读体验偏紧凑
- 用户在 Zotero 中调整全局字体大小时，Deepseek Copliot 不容易自然跟随
- 后续如果继续优化字号，需要在多个文件里重复修改零散字面量

与此同时，Zotero 自身已经把字体大小视为宿主级能力：

- 官方知识库明确提供 `View -> Font Size` 与 `View -> Note Font Size`
- hidden preferences 中存在 `extensions.zotero.fontSize` 与 `note.fontSize`
- Zotero 源码通过 `Zotero.UIProperties.registerRoot(root)` 统一给根节点下发字体倍率和相关属性

结合当前插件挂载方式来看，Deepseek Copliot 的主侧边栏本身就运行在 Zotero 已管理的 pane 容器里，因此理论上是有条件尽量继承宿主字体效果的。当前离“更原生”的主要阻碍，不是挂载位置，而是插件内部继续大量使用绝对字号和自定义字体族覆盖。

因此，这一轮更合适的方向是：以“原生优先”为原则，尽量继承 Zotero 宿主字体与字体倍率，减少硬编码字号，只在必要处保留少量相对层级控制。

## 范围

本次修改包括：

- 将 Deepseek Copliot 主要侧边栏体验中的字体策略改为“宿主继承优先”
- 尽量移除主要 UI 中的绝对 `px` 字号，改用继承字体大小加少量相对单位表达层级
- 视情况移除不必要的自定义字体族覆盖，让主侧边栏更贴近 Zotero 原生字体观感
- 在保留层级关系的前提下，微调行高与间距，避免改成继承后界面显得松散或拥挤
- 将项目版本从 `0.9.0` 升级到 `0.9.1`

本次修改不包括：

- 新增用户可选的 `小 / 中 / 大` 字体设置
- 新增一套插件私有的字体倍率系统
- 直接重建一整套完整设计 token 系统
- 更大范围的视觉改版
- 修改与可读性无关的功能、文案或交互行为
- 主动调整 Zotero 原生设置页控件的字体，除非发现某个明显属于 Deepseek Copliot 自己控制的元素仍然过小

## 核心决策

`v0.9.1` 采用“原生优先、继承宿主、少量相对层级”的方案。

这意味着：

- Deepseek Copliot 优先吃 Zotero 宿主提供的字体与字体倍率
- 插件不再依赖大量绝对 `px` 字号来维持布局
- 只有标题、meta、badge、代码块、表格等确实需要层级表达的地方，才保留少量相对缩放

之所以优先选这条路，是因为：

- 用户的真实诉求不只是“再大一点”，而是“更像 Zotero 本身”
- Zotero 已经有自己的全局字体调整心智和实现路径
- 当前代码里最大的问题是插件自己把宿主继承链切断了
- 相比继续调一批 `px` 数值，回到宿主继承模型更稳，也更利于长期维护

## 设计

### 1. 目标表面

优先调整 Deepseek Copliot 自己掌控、且构成日常使用主路径的文本区域：

- 侧边栏头部与其元信息
- 当前 scope 摘要区域
- notice、badge、提示条
- 最近线程列表与线程操作按钮
- 空状态文案
- composer 输入框、辅助文案与预设菜单
- 聊天消息渲染区域，包括 markdown 标题、列表、表格与代码块
- 选中文本后的 popup 按钮，以及仍然明显偏小的 Deepseek Copliot 自有标签

本轮不把 Zotero 原生设置页控件作为主要优化目标。设置页大多继承宿主样式，并不是这次“字体偏小”反馈的主要来源。

### 2. 字体策略

核心策略不是继续“统一调大一档”，而是优先回到宿主继承模型：

- 主容器尽量不再显式指定 `font-size`
- 主容器尽量不再覆盖宿主字体族，除非某个局部表面确实需要特殊字体
- 正文、输入文本、常规列表文案优先直接继承宿主字号
- 次级说明、meta、badge、按钮文案等，使用 `em` 或其他相对单位表达层级
- markdown 标题层级保留，但改为相对正文缩放，而不是固定 `16px / 15px / 14px`

推荐的整体思路如下：

- 正文层：`inherit` 或 `1em`
- 次级说明层：略小于正文，例如 `0.92em` 到 `0.96em`
- 极小标签层：不低于当前 Zotero 观感底线，避免继续出现“读起来发虚”的 `10px`
- 标题层：基于正文做相对放大，例如 `1.08em`、`1.16em`、`1.24em`

在这个方案下，如果用户通过 Zotero 的 `View -> Font Size` 调整全局界面字体，Deepseek Copliot 的大部分文本就会更自然地一起变化。

### 3. 代码结构

这一轮不以“新增一套字号 token”为目标，而以“移除不必要的绝对字号”为目标。

如果在实现过程中仍然需要一些统一层级，优先引入语义化、相对化的层级命名，而不是继续固化为新的绝对像素值。例如：

- `body`
- `meta`
- `label`
- `heading-sm`
- `heading-md`

这些层级如果存在，也应优先表达“相对关系”，而不是隐藏一批新的 `px` 常量。

这次实现的重点预计仍在 React UI 层：

- `src/ui/components/Sidebar.tsx`
- `src/ui/components/Composer.tsx`
- `src/ui/components/ThreadView.tsx`
- `src/ui/components/EmptyState.tsx`
- `src/ui/components/ScopeBar.tsx`

次级检查目标：

- `addon/content/styles.css`
- `src/modules/readerIntegration.ts`
- `src/ui/ui.ts`，前提是其中仍有会影响最终交付体验的旧字号实现

### 4. 原生对齐原则

这次实现需要遵守以下原生对齐原则：

1. 能继承宿主的地方，不主动覆盖。
2. 能用相对单位表达层级的地方，不写绝对 `px`。
3. 能删掉多余字体族声明的地方，不保留视觉上“像 Web App 而不像 Zotero”的自定义字体栈。
4. 只有在 popup、代码块、表格、特殊标签这类明确需要局部控制的表面，才保留有限样式约束。

### 5. 版本策略

这次工作归入 `v0.9.1`。

对外发布版本应从 `0.9.0` 升级到 `0.9.1`，并保持现有基于 `package.json` 派生出来的构建与发布链路继续正常工作。

## 验收标准

如果满足下面这些条件，就可以认为 `v0.9.1` 的字体可读性优化达标：

1. Deepseek Copliot 在 library 与 reader 两种侧边栏场景下，都比当前版本更接近 Zotero 原生字体观感
2. 主正文和常规交互文本大多依赖宿主继承，而不是依赖插件内部绝对 `px` 字号
3. 仍然需要层级区分的地方，使用少量相对缩放即可表达清楚，不再零散打补丁
4. 改成继承后，没有出现明显的截断、重叠、过度松散或控件拥挤
5. markdown 消息渲染仍然保留清晰的标题层级，并且表格、代码块可读
6. 没有新增字体大小设置项
7. 项目版本号为 `0.9.1`

## 风险与缓解

### 风险：完全去掉绝对字号后，层级变得不够清楚

缓解方式：

- 保留少量相对层级，而不是追求“零样式”
- 继续通过字重、间距和相对倍率拉开标题与元信息

### 风险：继承宿主后，不同挂载面的观感略有差异

缓解方式：

- 在 library 和 reader 两种挂载面都做验证
- 只对确实失衡的局部表面做轻量补偿，而不是重新回到全局硬编码

### 风险：局部 popup 或特殊区域仍然遗留绝对字号

缓解方式：

- 把 popup、badge、label 作为次级检查面专门过一遍
- 对不能完全继承的表面，也优先使用相对单位

### 风险：为了“原生”反而把代码改得过重

缓解方式：

- 这次只处理 Deepseek Copliot 自己控制的表面
- 不扩展到完整主题系统或宿主偏好深度联动

## 实施备注

这次应保持为一次小而稳的 polish 版本。如果实现过程中开始演变成完整的字体配置系统、与 Zotero 全局字体偏好的深度联动机制，或者更广泛的侧边栏改版，就应当停止扩展范围，并把那部分拆成 `v0.9.1` 之后的独立项目。
